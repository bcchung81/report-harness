# KCA 보고서 레이아웃·편집 규율 (SSOT)

개조식 본문이 **읽는 사람 기준으로 완결**되도록 다듬는 규칙의 단일 기준. 문체 규칙(명사형·2줄)은 `style-rules.md`, 이 스킬은 **분량·계층 밀도·용어 접근성·도해**를 담당한다. `kca-writer`가 작성 시 적용하고, `kca-style-auditor`가 `kca-style-lint`의 L11~L15로 검증한다.

> **SSOT 역할 분담(중복 방지):** 규칙의 *임계값·방법*은 **이 스킬(layout)이 유일 원천**이다. `kca-style-lint`는 그 규칙을 *탐지·판정하는 체크리스트*(L11~L15)이며 임계값은 이 스킬을 인용한다. 규칙을 바꿀 때는 **이 스킬을 먼저 고치고** lint·auditor·builder는 참조만 갱신한다(사본을 각자 두지 말 것).

## 1. 분량 예산 — 본문은 2페이지 이내(최대 3)

공공기관 보고서는 **한눈에 요지가 잡혀야** 한다. 본문(주요 내용)은 **2페이지 이내 종료를 목표**, 불가피할 때만 3페이지까지.

- 본문에는 **의사결정에 필요한 핵심만** 남긴다: 배경 요지·핵심 추진 내용·기대효과·일정.
- 근거 상세·세부 데이터·부연 설명·긴 표·참고자료는 **본문에서 덜어내 「붙임」으로 이관**한다.
- **판단 기준**: "이 항목이 없으면 의사결정이 막히는가?" → 아니오면 붙임 후보.

### 페이지 어림 규칙 (개조식 15pt 기준)

한 페이지 ≈ □ 3~4개 + 각 □당 ㅇ 2~3개(각 2줄) + 표 0~1개. 즉 **본문 총 ㅇ 12~18개 + 표 1~2개**가 2페이지 상한의 현실적 눈금. 이를 넘으면 통합·붙임 이관을 검토한다.

## 2. □당 ㅇ 최대 3개 — 계층 밀도 제어

하나의 `□`(섹션·대주제) 아래 `ㅇ` 항목은 **최대 3개**. 4개 이상이면 독자가 요지를 못 잡는다.

- 3개를 넘으면 우선순위로 **핵심 3개만 본문**에 남기고 나머지는 (a) 유사 항목끼리 **통합**하거나 (b) **붙임**으로 이관한다.
- `ㅇ`가 통합·삭감돼도 정보는 하위 `-`나 붙임에 보존한다(내용 손실 금지).

## 3. ㅇ 통합 규칙 — 중복·유사 항목 병합

작성된 `ㅇ` 중 **주제가 겹치거나 인접한 것**은 하나로 통합한다.

- **통합 신호**: 같은 대상(예: 두 ㅇ 모두 "에이전트 신설"), 원인-결과 관계, 수단-목적 관계, 같은 시점 일정.
- **통합 방법**: 상위 개념을 괄호 리드로 세우고, 세부 차이는 하위 `-`로 내린다.
  - ❌ 분산: `ㅇ (문체 루프) writer⇄auditor …` / `ㅇ (렌더 루프) builder⇄qa …`
  - ✅ 통합: `ㅇ (생성-검증 루프) 작성자≠검증자 분리 원칙의 2개 루프 도입` → 하위 `-` 문체 루프 / `-` 렌더 루프
- 통합으로 `□당 ㅇ ≤ 3` 을 달성한다. 통합의 목적은 삭제가 아니라 **계층 재배치**다.

## 4. 용어 각주 `*` — 비전문가 접근성

`ㅇ`·`-` 본문의 단어 중 **비전문가가 이해 불가하거나 IT·전문용어**인 것은 각주 처리한다.

- **식별 기준**: 영문 약어(RAG·LLM·sLLM·PoC·ROI·API), 기술 개념(온프레미스·파이프라인·오케스트레이터·charPr·SSOT), 제품·규격명(H200), 조직 전문어. 일반 행정 용어(추진·정립·고도화)는 제외.
- **표시**: 본문에서 해당 단어 **바로 뒤에 `*`** 를 붙인다(예: `온프레미스* 환경`). 같은 용어가 여러 번 나오면 **첫 등장에만** `*`.
- **⚠️ ㅇ당 각주 마커 최대 1개** — 한 `ㅇ` 항목 아래 `*`는 **1개만**. 한 ㅇ에 각주 대상 용어가 여럿이면:
  1. **우선순위**로 가장 중요한 키워드·필수 정보 **1개만** `*` 표시, 나머지는 **2줄 본문에 녹이거나 생략**(과다 각주 금지).
  2. 그래도 **의미 전달이 꼭 필요한 보충**이 있으면 `*` 대신 **당구장 기호 `※`로 별도 1줄**(ㅇ 아래). `※`도 **ㅇ당 최대 1개**, 글꼴은 `*`와 동일 각주 폰트(맑은고딕 12pt·charPr49) 유지.
  - 즉 한 ㅇ 아래 각주 마커 총량 = `*` ≤1 + `※` ≤1. `ㅇ` 하위에 `*`를 2개 이상 달지 않는다.
- **정의 요약**: 본문 블록 하단(또는 붙임 직전)에 `* 용어: 한 줄 정의`로 모아 정리. 정의는 **1줄·명사형·평이한 말**로.
  ```
  * 온프레미스(On-Premise): 외부 클라우드가 아닌 기관 내부 서버에 직접 구축·운영하는 방식
  * RAG: 외부 문서를 검색해 답변 근거로 쓰는 생성형 AI 기법
  ```
- **정의 날조 금지**: 뜻이 불확실하면 지어내지 말고 `firecrawl:firecrawl-search`(일반 용어)·`Context7`(라이브러리·제품)로 확인하고, 검색이 차단되면 `insane-search`로 우회한다. 다출처 대조가 필요하면 `deep-research`를 쓴다. 확인 불가 시 `[정의 확인 필요]`로 남긴다.
- **⚠️ 렌더 주의(builder 처리)**: 하단 정의 줄의 선두 `*`는 마크다운 불릿이라 kordoc이 `□` 표제로 오매핑한다. builder가 generate_document 전에 `* ` → 전각 `＊ `로 치환해 각주 스타일(맑은고딕12)로 렌더한다(writer는 `*`로 작성, 변환은 builder 소관).

## 5. 붙임(별첨) 구조

본문에서 덜어낸 상세를 담는다. 본문 뒤에 배치.

```
────────────────  (본문 끝)

붙임 1. {제목}
  {상세 표·데이터·부연 — 개조식 유지}

붙임 2. {제목}
  {참고자료·근거 목록}
```

- 붙임은 **본문 페이지 예산에 포함하지 않는다**(붙임은 조회용).
- **⚠️ 붙임은 최소 1페이지를 채운다** — 붙임을 둘 거면 내용이 **한 페이지 이상**이 되도록 상세·표·부연을 충분히 담는다. 붙임 여러 개(붙임 1·2·3…)를 두더라도 **합쳐서 최소 1페이지**를 채우도록 작성한다. 반 페이지짜리 앙상한 붙임은 본문에 흡수하거나 삭제한다(붙임은 "덜어낸 상세를 제대로 담는 곳"이지 자투리 보관함이 아니다).
- 각 붙임은 `붙임 N. 제목` 한 줄 표제로 시작. **렌더 규약(실물 KCA 보고서 실측)**: builder가 이 표제를 「붙임 N」 **라벨박스 표 헤더**로 변환한다 — 1행 3열(라벨 5968/스페이서 565/제목 41626), 라벨 셀은 4방 0.5mm 테두리 + 진남색(#2B2D63) 음영 + 흰 글자 16pt HY헤드라인M, 제목 셀은 상하선만 0.5mm + 16pt. 각 붙임은 새 페이지에서 시작(pageBreak). writer는 MD에 `붙임 N. 제목` 텍스트만 쓰면 된다(변환은 builder 소관).
- **⚠️ 붙임 본문 계층 구조 (필수)** — 붙임은 표만 던져놓는 자료 창고가 아니다. 각 붙임은 라벨박스 헤더 아래 다음 계층으로 작성한다:
  1. **`ㅇ` 요약 설명(2줄 이내, 1개 이상)** — 이 붙임이 무엇을 보여주는지·핵심 결론이 무엇인지 개조식 요약. 표보다 먼저 온다.
  2. 추가 설명 필요 시 **`-` 세부**(계층형, 2줄 이내).
  3. 그 하위 부연 필요 시 **`*`(용어·수치 각주)·`※`(단서·부연)** 계층 — 각주 폰트(맑은고딕 12pt) 유지, ㅇ당 `*`≤1+`※`≤1 규칙은 붙임에도 동일 적용.
  4. 그 다음 `[ 캡션 ]` + 표·데이터.
  MD 표기: `- 붙임 N. 제목`을 최상위 리스트로 두고 `  - (리드) 요약`(→ㅇ), `    - 세부`(→-)를 중첩, `*`·`※`는 리스트 밖 평문 줄로.
- 본문에서 "(상세) 붙임1 참조"처럼 **포인터**로 연결한다.
- HWPX 변환 시 붙임은 본문 뒤 별도 문단군으로 렌더(kordoc). 표는 붙임에서도 `[ 캡션 ]` + GFM 표 규칙 유지.

## 6. 워크플로우 도해 (SVG → PNG → HWPX 주입)

보고서에 **처리 흐름·조직도·파이프라인**을 그림으로 넣어야 할 때의 표준 경로. HWPX(kordoc)는 이미지를 못 만들고 표·텍스트만 렌더하므로, 아래 4단계로 직접 그린 SVG를 PNG로 바꿔 주입한다. 외부 의존(Figma·plan key) 없이 로컬 도구만 사용.

| 단계 | 도구 | 비고 |
|------|------|------|
| ① SVG 작성 | 직접 코드 작성 | `claudian/attachments/*.svg` 스타일(둥근 박스·화살표 마커·`Noto Sans KR`) 재현 |
| ② SVG→PNG | `qlmanage -t -s 2000 -o <dir> <svg>` (macOS 네이티브) | 설치 불필요 |
| ③ 크롭·여백 | Pillow — 비-흰색 bbox 크롭 + 균일 흰 여백 | 정사각 패딩 제거 |
| ④ HWPX 주입 | `scripts/inject_image.py <base.hwpx> <png> <out.hwpx> --anchor "텍스트" --width-mm 170` | BinData 추가 + manifest 등록 + 앵커 문단 뒤 그림 삽입 |

**⚠️ 치명적 제약 — qlmanage는 SVG 캔버스를 정사각에 맞춘다.** 가로로 긴 워크플로우 SVG를 그대로 래스터하면 **오른쪽이 잘린다**(실측 확인). 반드시 **SVG 캔버스를 정사각(예: `width="1120" height="1120" viewBox="0 0 1120 1120"`)으로 만들고 도해는 상단에 배치**한다. 하단 흰 여백은 ③에서 크롭된다. 가로 캔버스로 두면 클리핑.

**주입 규칙**:
- `inject_image.py`는 `content.hpf` manifest에 `imageN`(연속 id)으로 등록하고 `hp:pic`(treatAsChar 인라인, horzAlign=CENTER)을 삽입한다. HWPX 스타일 id 불변식은 이미지에 영향 없음(이미지는 manifest opf:item으로만 등록).
- `--anchor "추진 내용"`처럼 도해를 넣을 섹션의 표제 텍스트를 지정한다. 생략 시 본문 끝.
- `--width-mm`은 표시 폭(기본 170mm ≈ A4 본문 폭). 세로는 원본 비율로 자동.
- 주입 후 **반드시 `validate_hwpx.py`(구조) + `parse_document`(내용)로 재검증**한다. parse 결과에 `![image](...)`가 앵커 위치에 나타나면 성공.

**언제 도해를 쓰나**: 5단계 이상 흐름·분기·피드백 루프가 있어 텍스트 화살표(`⇨`)로는 관계가 안 보일 때. 단순 순차 흐름은 개조식 `A ⇨ B ⇨ C`나 단계 표로 충분(도해 남용 금지 — 본문 분량 예산에 그림도 포함).

## 7. 글 품질 규율 — 구조를 넘어 "잘 쓴 글"로

구조 규칙(명사형·2줄)을 지켜도 글이 죽을 수 있다. gold 코퍼스 실측 대조에서 확인된 4대 품질 축:

### 7-1. 향후계획 일정 필수 (P1)
향후계획의 각 `ㅇ`는 **반드시 시점을 병기**한다: `항목 : '26. M월 중` / `'26. M월 ~ M월`. 일정이 미확정이면 지어내지 말고 `['26.__월]` 플레이스홀더 + "(안)"으로 남긴다. **날짜 없는 향후계획은 계획이 아니라 선언문이다** — 공공보고서 향후계획의 본질은 일정 약속.

### 7-2. 구체성 예산 (P2)
각 `ㅇ`에 **수치·날짜·고유명(시스템명·조직명) 중 최소 1개**를 담는다. 자료가 없으면 추상 명사구("운영 기반 정립")로 위장하지 말고 플레이스홀더 `[○○건]`로 노출한다 — 빈 곳이 보여야 사용자가 채운다. gold 기준 수치 밀도는 1000자당 ~4.5토큰; 절반 이하면 추상어 과다 신호.

### 7-3. 중복 상한 (P3)
동일 핵심 문구는 **본문+붙임 합산 3회 이하**. 붙임은 본문 문장을 재사용하지 않는다 — 붙임의 존재 이유는 반복이 아니라 **심화**(데이터·근거·상세)다. 같은 개념을 다시 말해야 하면 다른 층위(수치·사례)로 내려가라.

### 7-4. 리듬 다양화 (P4)
`ㅇ` 구문 골격을 순환한다 — 같은 연결 패턴("~해,"·"~로,")이 **연속 3항목 이상 반복 금지**. 세 가지 골격을 섞는다:
- **단문형**: `(리드) 핵심 명사구 + 마감어` — 짧고 단정
- **인과형**: `(리드) 원인/배경에 따른 결과 + 마감어`
- **대비형**: `(리드) 기존 A에서 B로 전환 + 마감어`

### 7-5. 30초 테스트 (P5)
탈고 전, **□ 표제 + 괄호리드만** 추출해 읽는다. 의사결정자가 30초 안에 (a) 무엇을 했고 (b) 무엇을 결정/승인해야 하는지 파악되면 통과. 안 되면 리드가 장식이라는 뜻 — 리드를 정보어로 교체한다.

### 7-6. 핵심 메시지 선행 — BLUF (P6)
**결론·건의를 본문 맨 앞에 세운다(Bottom Line Up Front).** 배경부터 쌓아 올려 마지막에 결론이 나오는 미괄식은 1페이지 보고서의 적이다 — 바쁜 의사결정자는 첫 화면에서 "무엇을 승인해달라는 것"인지 알아야 한다. 근거는 [[one-pager-principles]](두괄식·Minto 피라미드·아마존 answer-first가 공통으로 지지).

- **배치**: 골격(§7 L7의 추진배경·추진내용·향후계획)을 훼손하지 않고, **추진배경 앞에 선두 블록 `□ 보고 요지`(또는 `□ 건의 요지`)를 1개** 둔다. 그 아래 `ㅇ` **1~2개**로 핵심 메시지·건의를 개조식 명사형으로 압축한다(각 2줄).
  ```
  □ 보고 요지
    ㅇ (핵심) {무엇을 왜 한다/건의한다 — 결론 한 문장을 명사형으로}
    ㅇ (건의) {의사결정자에게 구하는 승인·조치 — 없으면 생략}

  □ 추진 배경 및 목적
    …
  ```
- **핵심 메시지 = 피라미드 정점(governing thought)**: 이 선두 블록의 `ㅇ`는 이어지는 본문 전체를 한 문장으로 요약해야 한다. 본문 근거들이 이 문장을 떠받치는 구조(§2 □당 ㅇ≤3 = 피라미드 3근거와 정합)여야 한다.
- **§7-5와의 구분**: 30초 테스트(7-5)는 *다 쓰고 나서* 리드만 읽어 판단되는지 **점검**하는 규칙이고, 이 7-6은 결론을 물리적으로 최상단에 **배치**하는 구조 규칙이다. 7-6으로 배치하고 7-5로 검산한다.
- **대상별 강제 수위(§9 연동)**: **기관장·임원 = 필수**(선두 블록 없으면 반려). 경영진 = 권고. 부서장·실무 = 권고(실행 상세형이면 배경 선행도 허용). 검증은 `kca-style-lint` **L22**.
- **날조 금지**: 건의·결론이 입력자료에 없으면 지어내지 말고 planner에게 확인받거나 `[건의 확인 필요]`로 노출한다.

## 8. 표 활용 — 서술 대신 표 (본문·붙임 공통)

**비교·나열·대응 관계는 문장으로 풀지 말고 표로 정리한다.** 본문에서도 다음 신호가 보이면 표로 전환한다:
- 같은 속성 축(구분/내용/시점 등)으로 3개 이상 항목을 나열할 때
- 전·후 비교, 기관·모델·정책별 대비, 일정·단계 매트릭스
- `ㅇ`·`-` 서술이 사실상 "A는 x, B는 y, C는 z" 병렬 구조일 때
표는 `[ 캡션 ]` + GFM 표로 작성하고, 본문 분량 예산(2p)에 포함한다(표 남용 금지 — 요지 전달에 표가 더 빠를 때만).

**본문 표 크기 규율 (반 페이지 상한)**: 본문 표가 반 페이지를 넘으면 본문 2p 예산이 무너진다. **원본 데이터(제공 자료·조사 결과)를 전량 나열하지 않는다** — 본문 표는 다음 두 형태만 허용:
- **핵심 행 선별**: 의사결정에 직결되는 대표 row만 남기고 마지막 행에 `외 N건 (상세 붙임M)` 표기
- **통계 가공**: 원본을 집계 형태로 변환 — 주체별·연도별·유형별 건수, 추이, 상위 N. 원본 전량은 붙임 총괄표로 이관
- **어림 눈금(SSOT)**: 본문 표 1개당 **데이터 행 ≤ 8**(헤더·구분선 제외). 셀이 2줄로 긴 표는 그보다 적게. 검증은 lint **L23**(기계 판정 — `lint_report_md.py` `BODY_TABLE_ROWS_MAX`). 붙임 표는 행 수 제한 없음(조회용).

**열 폭 (자동 재분배 + 지시자)**: kordoc은 열 폭을 균등 분할하므로 builder가 `adjust_table_widths.py`로 셀 내용 표시폭(한글 2·영숫자 1)에 비례해 자동 재분배한다 — 하한 열당 전체 폭 8%·헤더 폭 보장, 가중치 상한 5:1, 표 전체 폭 보존. writer가 특정 비율을 원할 때만 캡션 지시자 `[ 캡션 | 폭 2:1:1:3 ]`(비율 개수 = 열 수)를 쓴다. 지시자는 렌더 전에 캡션에서 제거되고 비율만 적용되며, 형식 오류·개수 불일치 시 자동 분배로 폴백한다(빌드 중단 없음).

**표 안 글꼴 규정(사용자 지정 '26.7.13)**: 표 내부 텍스트는 **맑은 고딕 12pt** — 헤더 셀 12pt Bold(+#FFF7CC 음영), 본문 셀 12pt. (이전 KoPub돋움체 12/11에서 변경. 상세는 `style-spec.md` SSOT.)

## 9. 보고 대상 프로파일 — 보고자 수준 맞춤 (SSOT)

보고서 생성 시작 시 오케스트레이터가 **보고 대상 1개**를 선택받아 `{REPORT_DIR}/00_context.md`에 기록한다(자료 구성 방향과 함께). 아래 4개 축이 §1 분량 예산 등 기본값을 **대상별로 오버라이드**하는 유일 기준이다. planner는 관점 축에 맞춰 재료를 모으고, writer는 해당 열을 준수하며, style-auditor가 lint L21로 검증한다.

| 축 | 임원 | 경영진 | 부서장·실무 |
|---|---|---|---|
| 분량·구성 | 본문 1p 목표(최대 2p), 결론·건의 선행 | 본문 2p, 전략+관리 균형 | 본문 2~3p, 실행 상세 |
| 용어·각주 | 전문용어 → 쉬운 표현 치환 우선, 필수 용어만 각주 | 핵심 용어만 각주 | 전문용어 허용, 각주 최소 |
| 관점·메시지 | 의사결정·예산·리스크 | 전략·성과·조직 | 실행계획·일정·담당 |
| 표 상세도 | 총괄표만 본문, 상세 표는 붙임 | 총괄표+핵심 비교표 본문 | 상세 표 본문 허용 |

- 각주 총량 규칙(ㅇ당 `*`≤1+`※`≤1)은 모든 프로파일 공통 — 임원 프로파일은 "각주 증가"가 아니라 "쉬운 표현 우선"으로 해석한다.
- `00_context.md` 없이 실행되면 대상을 되묻는다(기본값 가정 금지).

## 10. 법령 인용 표기 — 한국 공문서식 (SSOT)

법령 근거는 **법제처 공문서 표기 관례**를 따른다. 독일식·학술식 표기 금지.

- **법령명**: 첫 인용은 「정식 법령명」 전체(예: 「방송통신발전 기본법」), 이후 약칭을 쓰려면 첫 인용에 `(이하 "방송통신발전법")` 선언 후 사용. "방발"·"정진" 같은 미선언 임의 약칭 금지.
- **조항호목**: `제N조제N항제N호N목` — **전부 한글·붙여쓰기**. 가지조문은 `제N조의2`.
  - ✅ `제66조제1항`, `제66조제4항제5호`, `제45조의2제3항`
  - ❌ `§66①`(독일식), `제66조①`(원문자 항), `제66조 제1항`(띄어쓰기), `Art. 66`
- **결합 인용**: `「전파법」 제66조제1항·제4항제5호`처럼 법령명은 한 번, 조항은 가운뎃점 나열.
- 검증은 lint **L24**(기계 판정 — `§`·조 뒤 원문자·조항 띄어쓰기 탐지). planner가 korean-law MCP로 조문 확인 시점부터 이 표기로 수집한다(writer 단계 변환 금지 — 수집 원문 그대로 흐르므로 첫 기록이 표준이어야 함).

## Why — 왜 이 규율인가

개조식의 목적은 **의사결정자가 30초 안에 요지를 잡는 것**이다. `ㅇ`가 무한정 나열되면(현행 문제) 개조식의 장점이 사라진다. 밀도 제어(□당 ㅇ≤3)·분량 예산(2p)·용어 접근성(`*`)·상세 격리(붙임)는 모두 "본문은 요지, 상세는 붙임"이라는 한 원리의 구현이다.
